Customer smoke test

Pre-launch end-to-end check. Walks through sign-up, onboarding, ticket creation, profile, uploads, and feedback on the live brandbite.studio environment. Mark each step Pass / Fail / Skip and add notes for anything weird. Your progress is saved in this browser; you can come back later.

Before you start:
  • Use a brand-new email you haven’t signed up with before (Gmail + addressing works: you+ct1@gmail.com).
  • Have a 1–2 MB sample image (JPG or PNG) and a ~500 KB PDF ready to upload.
  • If your plan requires payment, have a test card or your real card ready.
  • Ideally also open brandbite.studio/admin in a different browser / incognito as your existing site-owner account so you can verify the admin side sees the events.

Phase 1 · Sign up

Brand-new account. We’re testing the BetterAuth + verify-email + Turnstile path that real customers will hit on launch day.

1.1 Visit brandbite.studio

Action: Open https://brandbite.studio in a fresh browser tab (or incognito).

Expected: Marketing landing page loads. Sign-in / Get-started CTAs are visible in the header. No console errors. The page renders within ~2 seconds on a normal connection.

1.2 Click Get started / Sign up

Action: Click the primary CTA in the header or hero. Should land on /login with the form in “Create account” mode.

Expected: Login page renders. Toggle pill shows “Sign in” / “Create account” with Create-account selected (or you can click it). Cloudflare Turnstile widget renders below the form (small puzzle challenge).

1.3 Submit sign-up form

Action: Enter name, fresh email, password (12+ chars with upper/digit/symbol). Wait for Turnstile to verify, then click “Create account”.

Expected: Page redirects to /verify-email?email=…. A “Check your inbox” screen tells you to click the link in the email.

1.4 Open verification email

Action: Check the inbox of the email you used. Open the verification email from Brandbite.

Expected: Email arrives within ~30 seconds. Subject line is sensible (“Verify your email” or similar). Body has a clear “Verify email” button + a fallback URL. Brand looks right (orange, wordmark).

1.5 Click verification link

Action: Click the “Verify email” button.

Expected: Browser opens. You’re auto-signed-in and redirected to /onboarding. (No re-prompt for password — the verification flow auto-signs-in by design.)

Phase 2 · Onboarding

First-touch UX for launch traffic. Creates a Company and (optionally) a plan.

2.1 Create company

Action: Fill in company name (e.g. “Acme Test Co”). Slug should auto-fill from the name and be editable. Optionally add a website. Submit.

Expected: Wizard advances to plan selection. Slug is unique (no collision error).

2.2 Pick a plan

Action: Select a plan from the list. Plans show name, monthly tokens, price.

Expected: Plan list renders with at least one option. Prices match what’s configured in Stripe (no $0.00 weirdness). Selecting one advances the wizard or starts a Stripe checkout flow.

2.3 Complete payment (if required)

Action: If the plan requires payment, complete Stripe checkout with a real or test card. (Use Stripe test card 4242 4242 4242 4242 with any future expiry and any 3-digit CVC if you’re in Stripe test mode.)

Expected: Stripe Checkout iframe renders cleanly. Successful payment redirects back to Brandbite. Token balance reflects the plan’s monthlyTokens on first ticket creation.

2.4 Land on dashboard

Action: After onboarding finishes, you should land on /customer/board.

Expected: Sidebar visible with Workspace / Billing / Account sections. Empty board state (TODO / IN_PROGRESS / IN_REVIEW / DONE columns all empty). Floating “Feedback” pill bottom-right.

Phase 3 · Profile setup

The personal-account hub we built recently. Touches Identity, Timezone, Notifications, and the Avatar upload path.

3.1 Open profile

Action: Sidebar → Account → Profile. URL should be /customer/profile.

Expected: Profile page loads with sections: Identity, Timezone, Notifications, Two-factor authentication, Active sessions, Delete account.

3.2 Update name + upload avatar

Action: Edit name. Click “Upload avatar” and pick a JPG / PNG ≤ 2 MB.

Expected: Name save returns success toast. Avatar uploads, circle preview updates instantly with the image.

3.3 Set timezone (Auto-detect)

Action: Click “Auto-detect” in the Timezone section.

Expected: Detected zone fills in (e.g. Europe/Istanbul), saves, the “Your browser thinks you’re in…” helper hint disappears.

3.4 Toggle a notification preference

Action: Disable the email channel for one of the events (e.g. Status changed). Reload the page.

Expected: Toggle stays off after reload (preference persisted).

3.5 Enable two-factor (optional but recommended)

Action: Enter your password → Set up → scan the QR with Google Authenticator / 1Password / Authy → type the 6-digit code. Save the backup codes shown at the end.

Expected: QR renders locally (/api/profile/two-factor/qr). Code verifies. Backup codes display once with a Copy button. Enabled badge appears.

Phase 4 · Create first ticket

The bread-and-butter customer flow. Includes the new doc-upload path (PDF / DOC / TXT alongside images) at the 4 MB cap.

4.1 Open new ticket form

Action: Click “New ticket” or similar CTA on the board.

Expected: Lands on /customer/tickets/new. Form has project picker, title, brief, job type, priority, tags, and an attachments section that says “Accepts images, PDF, DOC, DOCX, TXT. Max 4 MB per file.”

4.2 Attach a 1–2 MB image + a PDF

Action: Drag (or pick) the image you prepared. Then pick the PDF.

Expected: Both files appear in the attachment list with filename + size + MIME. No upload errors.

4.3 Try a 5 MB file (negative test)

Action: Try attaching a file larger than 4 MB.

Expected: Inline error appears immediately (“filename is 5.X MB. The current upload limit is 4 MB.”) without any network round-trip.

4.4 Fill brief, pick job type, submit

Action: Fill title, write a short brief in the rich-text editor, pick a job type that has a token cost, set priority. Submit.

Expected: Form submits, lands on the new ticket detail or board. Ticket appears in TODO. Token balance has decreased by the job type’s cost × quantity. Attachments are visible in the ticket’s brief section.

4.5 Verify auto-assign (or fallback)

Action: Refresh the ticket detail page after a few seconds.

Expected: If a creative on the platform has the matching CreativeSkill and is below their cap, the ticket shows an assigned creative. If no qualified creative exists, the ticket stays unassigned (FALLBACK in audit log) — that’s also a pass for this smoke test.

Phase 5 · Revision feedback

This phase requires a creative to actually submit work. If you’re testing solo with no creatives, ask the operator to wear that hat (or skip and come back when there’s a real revision).

5.1 Notification email arrives

Action: When the creative submits a revision, watch your inbox.

Expected: Email arrives within ~1 minute. Subject is sensible (“New revision on ticket #…” or similar). CTA links to the ticket.

5.2 View revision + leave a pin

Action: Open the ticket, click the latest revision’s asset. Click somewhere on the image to drop a pin. Type a comment, save.

Expected: Pin appears at the click position. Comment posts. Pin count badge updates.

5.3 Approve or request revision

Action: Use the ticket actions to either approve (move to DONE) or request a revision.

Expected: Status updates correctly on the board. Creative receives an email about the decision.

Phase 6 · Feedback widget

Submit a real test entry through the floating widget so it shows up at /admin/feedback.

6.1 Open feedback modal

Action: Click the orange / dark “Feedback” pill bottom-right of any page. Or hit Ctrl + / (or ⌘ + / on Mac).

Expected: Modal opens with type select + headline + details.

6.2 Submit a praise entry

Action: Type: “Smoke test — checking the widget works end to end”. Pick “This is great” type. Submit.

Expected: Toast: “Thanks — we got it.”. Modal closes.

6.3 Verify it lands on /admin/feedback

Action: Switch to your admin browser tab, refresh /admin/feedback.

Expected: The praise entry appears under “New” with your customer email + role + page URL the test was submitted from. Status chips count updates.

Phase 7 · Sign-in & sessions

Verify the 2FA challenge fires for the password sign-in path and that active sessions appear correctly.

7.1 Sign out

Action: Sidebar → sign out (or hit /api/auth/sign-out via wherever the menu lives).

Expected: Redirected to / or /login.

7.2 Sign in with email + password

Action: Enter your password. Submit.

Expected: If you enabled 2FA in 3.5, the page swaps to a TOTP input (“Two-factor authentication”). If not, you land on the board directly.

7.3 Verify TOTP / backup code

Action: Enter the 6-digit TOTP from your authenticator. Or click “Use a backup code instead” and try one of the codes you saved in 3.5.

Expected: Code verifies, real session issues, you land on the board. (Skip if you didn’t enable 2FA.)

7.4 Active sessions list

Action: Profile → Active sessions.

Expected: At least one session listed, badged “This device”. Browser + OS parsed from User-Agent. IP shown masked (1.2.x.x).
