Creative / talent smoke test

Pre-launch end-to-end check for the talent funnel and the creative experience. Walks through the public application, admin hiring, magic-link onboarding, ticket assignment, deliverables, withdrawals, and feedback. Phases with a CANDIDATE badge are done from the candidate / creative side; phases with ADMIN need you in your existing site-owner browser tab. Progress saved in this browser.

Before you start:
  • Use a fresh email for the candidate side (Gmail + addressing works: you+ct1@gmail.com).
  • Have a portfolio URL ready (any real one), and a 1–2 MB sample image to submit as a deliverable later.
  • Open brandbite.studio/admin in a different browser / incognito as your existing SITE_OWNER account so you can switch hats easily.
  • For the assignment phase you’ll need a real customer ticket waiting in the system — either pre-create one as a customer, or pair this run with the customer smoke test (do that one first up to phase 4).

Phase 1 · Public application Candidate

Anonymous form at /talent. This is what every prospective creative fills out first.

1.1 Visit /talent

Action: Open https://brandbite.studio/talent.

Expected: Application form renders. Brand looks right (orange wordmark up top). Categories load. Scroll: every section is reachable. No closed-state banner (unless you toggled the kill-switch on).

1.2 Fill the form

Action: Fill name, WhatsApp (with country code), email (fresh / the one you prepared), country, timezone (auto-fills if available), portfolio URL, optional LinkedIn, pick at least one category, set total years, answer the remote / workload / tools / turnaround questions, tick the communication confirm.

Expected: Inline validation behaves: empty required fields disable submit. Conditional sections appear (e.g. yearsRemote when hasRemoteExp = true).

1.3 Solve Turnstile + submit

Action: Wait for Cloudflare Turnstile to verify (or solve the challenge if it appears). Click Submit application.

Expected: Submit succeeds, page swaps to a thank-you state. No 429 / 400 errors.

Phase 2 · Admin: accept + offer slots Admin

Switch to your admin browser tab. The submission should be in the talent queue.

2.1 Open /admin/talent-applications

Action: In the admin browser, visit /admin/talent-applications.

Expected: Submission appears at the top of the New chip with the candidate’s name + email. Counts on the New chip increment by 1.

2.2 Open detail panel + Accept

Action: Click Open. Click Accept. Pick three interview slots in the next 7 days (use the datetime pickers). Optionally add a custom message. Confirm.

Expected: Status flips to AWAITING_CANDIDATE_CHOICE. Toast confirms slots offered. Status chip count moves from New to Awaiting candidate.

Phase 3 · Candidate books a slot Candidate

The candidate gets a tokenised booking link in their email. They click, pick one of the three slots (or propose their own time).

3.1 Slot-offer email arrives

Action: Check the candidate’s inbox.

Expected: Branded email with three time slots and a “Pick a slot” button. Sender + subject look professional.

3.2 Click booking link, pick a slot

Action: Click the button. On the booking page, pick the soonest slot.

Expected: Confirmation screen renders. A second email lands with the Google Meet link + Calendar invite.

3.3 Verify Google Calendar event

Action: Check Google Calendar (the brandbite owner’s calendar configured in env).

Expected: Event appears at the chosen slot with Meet link + both attendees (owner + candidate). Description includes candidate context.

Phase 4 · Admin: interview + hire Admin

Skip the actual interview for the smoke test — just walk through the status transitions.

4.1 Mark interview held

Action: In /admin/talent-applications, find the row (now under Accepted). Open detail. Click “Mark interview held”.

Expected: Status flips to INTERVIEW_HELD. Hire form appears.

4.2 Fill hire form + click Hire

Action: Working hours: 9-18 weekdays Europe/Istanbul. Approve the categories the candidate applied for (deselect any you don’t want). Set tasks/week cap (e.g. 3). Optional notes. Click Hire.

Expected: Status flips to HIRED. Banner now reads “onboarding pending”.

4.3 Click Onboard now

Action: Click the “Onboard now” button on the HIRED row.

Expected: Status flips to ONBOARDED. A new UserAccount(role=DESIGNER) is created with the candidate’s email. Two emails go out: a magic-link sign-in and a branded welcome. Toast confirms.

4.4 Spot-check the new user in /admin/users

Action: Visit /admin/users, filter by Creative, find the new row. Click the name to open /admin/users/[id].

Expected: Detail page shows Creative role + the working hours you set + the cap + the talent-application link + the contact info (WhatsApp, country, portfolio).

Phase 5 · Creative first sign-in Creative

Switch back to the candidate’s inbox. The magic-link email is the entry point.

5.1 Welcome email arrives

Action: Check inbox.

Expected: Two emails: “Welcome to the team” (brand) and a BetterAuth magic-link “Sign in to Brandbite”. Both look right.

5.2 Click magic link

Action: Click the “Sign in” button in the magic-link email.

Expected: Auto-signed-in. Lands on /creative/board. Sidebar shows Creative-specific navigation. Floating Feedback pill is visible bottom-right.

Phase 6 · Profile + workload Creative

Sanity-check that hire-time data was migrated to the creative’s editable profile, and that creative-only sections render.

6.1 Open /creative/profile

Action: Sidebar → Account → Profile.

Expected: Sections: Identity, Timezone, Workload (creative-only), Notifications, Two-factor, Active sessions, Delete account.

6.2 Workload pre-fill

Action: Look at the Workload section.

Expected: Working hours field is pre-filled with what the admin set in 4.2. Concurrent task cap shows the same number set during hire. Both fields editable.

6.3 Adjust working hours

Action: Change the working hours text to something else, click Save.

Expected: Toast confirms. Reload the page — new value persists. Verify in admin /admin/users/[id] that the change shows up there too.

6.4 Skills + categories visible

Action: In your admin browser, open /admin/users/[id] for the new creative.

Expected: Skills card lists every JobType under the categories you approved during hire. Hired-from card links back to the talent application.

Phase 7 · Receive + deliver a ticket Creative

Requires a customer ticket waiting in the system whose JobType matches one of the creative’s approved categories. Pre-create one if needed.

7.1 Trigger auto-assign

Action: As a customer, create a new ticket using one of the JobTypes the creative has skills for. Or grab an unassigned one and re-trigger.

Expected: Within seconds, the ticket is auto-assigned to your new creative. Email + in-app notification fire.

7.2 Open ticket on creative side

Action: As the creative, open the ticket from the board.

Expected: Ticket detail loads. Brief, attachments, and tags visible. Status TODO. Action to start work / submit revision visible.

7.3 Submit a revision (output)

Action: Move ticket to IN_PROGRESS, then submit a new revision with the sample image you prepared.

Expected: Image uploads (4 MB cap). Revision saves. Status moves to IN_REVIEW. Customer notified by email.

Phase 8 · Earnings + withdrawal Creative

Verify the ledger credits the creative for the work and the withdrawal flow works end-to-end.

8.1 Customer approves the revision

Action: Switch to the customer browser, approve the revision (move to DONE).

Expected: Ticket status DONE. Creative receives a ticket-completed email. Ledger credits the creative per the payout rule for the JobType.

8.2 Check creative balance

Action: As creative, sidebar → Balance.

Expected: Balance reflects the credit. Lifetime earnings + available balance both correct.

8.3 Request a withdrawal

Action: Sidebar → Withdrawals. Request the available balance (or test below the MIN_WITHDRAWAL_TOKENS threshold to verify the block).

Expected: If above min — PENDING withdrawal lands at /admin/withdrawals. If below min — clean error showing the required threshold.

Submit a real test entry from the creative side so it lands at /admin/feedback alongside the customer one.

Expected: Modal opens.

9.2 Submit a feature-request entry

Action: Pick “Idea for a new feature”. Subject: “Smoke test — creative side”. Details: anything. Submit.

Expected: Toast: “Thanks — we got it.”.

9.3 Verify in /admin/feedback

Action: Switch to admin browser, open /admin/feedback.

Expected: Both customer and creative entries visible. Each shows the submitter’s email + role + page URL the test was submitted from. Counts add up.
